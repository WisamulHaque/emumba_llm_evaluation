# LLM Evaluation Guidelines

This document defines the core evaluation areas for LLM-powered applications. The evaluation strategy depends on the type of application being built. We categorize LLM applications into **three main categories**, each with its own set of evaluation areas.

---

## Category 1: Simple LLM Applications with Data Sources

These are single-responsibility LLM applications that interact with an external data source to answer user queries. Examples include Chat with Documents (RAG), Chat with APIs, and Chat with Databases.

---

### 1.1 Chat with Documents (RAG)

Applications where the LLM retrieves information from a document corpus (PDFs, knowledge bases, vector stores) and generates a response grounded in that context.

#### Evaluation Areas

**I. Retrieval Accuracy**
Validate that the chunks of information retrieved from the document store based on the user query are relevant and accurate. Poor retrieval leads to poor generation regardless of LLM capability.

**II. Response Accuracy**
Ensure the response generated by the LLM is factually accurate when compared against the ground truth and the retrieved context. This measures whether the LLM correctly synthesizes the retrieved information.

**III. Faithfulness (Hallucination Detection)**
Verify that every claim in the generated response is supported by the retrieved context. The LLM should not fabricate information that is absent from the source documents, even if the fabricated information happens to be factually correct.

**IV. Intent Accuracy**
Ensure the generated response fulfills the actual intent behind the user's query. The response should address what the user is truly asking, not just match keywords from the query.

**V. Response Completeness**
Validate that the response covers all relevant aspects of the user's query and does not omit critical information that was available in the retrieved context.

**VI. Citation Accuracy**
If the application provides source references or citations, verify that the cited sources actually contain the information attributed to them and that citations point to the correct documents or passages.

**VII. Conversational Flow**
Validate that conversation history is properly maintained across turns. The application should correctly handle follow-up questions, references to previous messages, and contextual continuity.

---

### 1.2 Chat with APIs

Applications where the LLM interprets user queries, selects the appropriate API endpoint, constructs the correct parameters, and presents the retrieved data to the user.

#### Evaluation Areas

**I. API Selection Accuracy**
Validate that the correct API endpoint is selected based on the user's query. When multiple APIs are available, the LLM should consistently pick the most appropriate one for the task.

**II. Parameter Accuracy**
Ensure the parameters constructed for the API call are correct and match the requirements of the user's query. This includes data types, required fields, and logical correctness of values.

**III. Response Formatting**
Validate that the data retrieved from the API is presented to the user in a clear, readable, and well-structured format appropriate to the data type and the user's request.

**IV. Response Completeness**
Ensure that all relevant data returned from the API is surfaced to the user and no critical information is dropped or omitted during the response generation step.

**V. Error Handling**
Validate that the application gracefully handles API errors, timeouts, rate limits, and unexpected responses. The user should receive meaningful feedback rather than raw error messages or silent failures.

**VI. Privacy and Data Masking**
Ensure that no private, sensitive, or personally identifiable information (PII) is exposed in the response after data retrieval from the API. Sensitive fields should be appropriately masked or omitted.

**VII. Multi-Step Reasoning**
When a user query requires multiple sequential API calls to resolve, validate that the LLM correctly determines the sequence of calls, passes intermediate results accurately, and synthesizes the final answer correctly.

---

### 1.3 Chat with Databases

Applications where the LLM translates natural language queries into database queries (SQL or NoSQL), executes them, and presents the results to the user.

#### Evaluation Areas

**I. Query Generation Accuracy**
Validate that the generated database query (SQL/NoSQL) is syntactically correct and semantically matches the user's natural language request.

**II. Schema Understanding**
Ensure the LLM correctly identifies and uses the right tables, columns, relationships, and joins relevant to the user's query. Misunderstanding the schema is one of the most common failure modes.

**III. Result Accuracy**
Verify that the query results returned to the user match the expected output for the given query. This includes checking aggregations, filters, and sorting logic.

**IV. Response Formatting**
Validate that raw database results are transformed into a user-friendly format (tables, summaries, natural language) appropriate to the query type.

**V. Data Security**
Ensure the application prevents SQL injection attacks and enforces proper access controls. The LLM should not generate queries that access unauthorized tables or expose restricted data.

---

### 1.4 Common Evaluation Areas (Applicable to All Data-Source Applications)

These evaluation areas apply universally across all simple LLM applications that interact with data sources.

**I. Out-of-Scope Accuracy**
Validate that the application correctly identifies and handles queries outside its defined scope. When users ask questions beyond the application's domain, it should gracefully decline rather than hallucinate an answer.

**II. Guardrails Accuracy**
Ensure the application upholds community guidelines, ethical standards, and content policies. It should not generate harmful, offensive, biased, or misleading content under any circumstances.

**III. Prompt Injection Resistance**
Validate that the application rejects all prompt injection attacks and does not violate its defined principles, system instructions, or security boundaries when users attempt to manipulate the prompt.

**IV. Consistency**
Verify that the application produces consistent quality outputs for similar queries. Repeated queries with the same intent should not yield wildly different quality levels or contradictory answers.

**V. Latency Awareness**
While not a quality metric per se, evaluation should track response times to ensure the application meets acceptable performance thresholds for the use case.

---

## Category 2: Multi-Agent Applications

These are complex applications composed of multiple specialized agents, tools, and decision pathways. The LLM orchestrates across these agents to resolve a user query, often involving multi-step reasoning and tool usage.

#### Evaluation Areas

**I. Behavioral Accuracy (Journey Validation)**
Validate that the pathway or journey taken by the LLM through the agent graph is correct. Given a user query, the sequence of agents invoked and decisions made should follow the expected logical flow.

**II. Agent Accuracy**
Evaluate the output of each individual agent in the pipeline. Each agent should produce accurate results within its scope, and the final aggregated output should be correct.

**III. Tool Call Accuracy**
Validate that the correct tools are selected and invoked at each step. This includes verifying that the right tool is called with the right parameters and that fallback tools are used appropriately when primary tools fail.

**IV. Intent Resolution**
Analyze the user query alongside the tool calls and agent outputs to determine if the user's original intent was correctly understood and resolved throughout the entire agent chain.

**V. Task Adherence**
Validate that the end goal of the user's query is fully achieved. This is a holistic check on the final output to confirm it directly and completely addresses what the user asked for.

**VI. Memory and Context Preservation**
Ensure that context is maintained and no information is lost as the user query passes through each agent in the pipeline. Intermediate results and conversation history should be preserved across agent handoffs.

**VII. Orchestration Efficiency**
Evaluate whether the agent orchestration follows an optimal path. Redundant agent calls, unnecessary loops, or suboptimal routing through the agent graph should be flagged.

**VIII. Error Recovery and Fallback Handling**
Validate how the system behaves when an individual agent fails or returns an unexpected result. The orchestrator should have fallback mechanisms and should not cascade failures silently.

**IX. Loop Detection**
Ensure the system detects and prevents infinite loops or circular dependencies between agents. A stuck multi-agent loop should be caught and terminated gracefully.

**X. Inter-Agent Data Integrity**
Verify that information passed between agents is accurate and complete. Data should not be lost, corrupted, or misinterpreted during handoffs between agents.

---

## Category 3: Simple LLM Applications without Ground Truth

These are applications where no external data source or ground truth exists. The LLM generates responses based solely on its training knowledge and the application's system prompt. Examples include conversational chatbots, interview-taking applications, coaching bots, and creative assistants.

#### Evaluation Areas

**I. Tonality**
Validate that the tone, style, and language of the application's responses align with the defined purpose and persona of the application. An interview bot should sound professional, a coaching bot should sound supportive, etc.

**II. Response Accuracy**
Ensure the application's responses are factually sound and align with the rules, instructions, and constraints defined in the system prompt. Even without ground truth data, responses should not contradict the application's own guidelines.

**III. Persona Consistency**
Verify that the application maintains its defined character, role, and personality consistently throughout the conversation. It should not break character or contradict its own prior behavior.

**IV. Coherence**
Validate that individual responses are logically structured, well-organized, and easy to understand. Responses should not contain contradictions, non-sequiturs, or disjointed information.

**V. Conversational Flow**
Evaluate the naturalness of the conversation. The application should handle topic transitions smoothly, maintain context across turns, and support natural turn-taking patterns.

**VI. Engagement Quality**
Assess whether the application's responses promote meaningful continued interaction. For applications like interview bots, this means asking relevant follow-up questions; for coaching bots, it means providing actionable guidance.

**VII. Empathy and Emotional Intelligence**
For applications that interact with users in sensitive contexts (interviews, coaching, support), validate that the responses demonstrate appropriate empathy, emotional awareness, and sensitivity.

**VIII. Out-of-Scope Accuracy**
Validate that the application correctly identifies and handles queries outside its defined domain. It should redirect the user back to the application's purpose rather than generating irrelevant or hallucinated responses.

**IX. Guardrails Accuracy**
Ensure the application upholds community guidelines and does not generate harmful, offensive, or inappropriate content regardless of user input.

**X. Prompt Injection Resistance**
Validate that the application rejects all attempts to manipulate its system prompt, override its instructions, or extract its internal configuration through adversarial user inputs.

---

## Summary

| Category | Key Focus | Primary Challenge |
|----------|-----------|-------------------|
| **Simple LLM Apps (with Data Sources)** | Accuracy of retrieval and response generation | Grounding responses in external data |
| **Multi-Agent Applications** | Orchestration correctness and tool usage | Maintaining coherence across agent chain |
| **Simple LLM Apps (without Ground Truth)** | Behavioral alignment and persona adherence | Evaluating without reference answers |
